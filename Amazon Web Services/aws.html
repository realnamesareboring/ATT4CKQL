This detection identifies EC2 instances where the IMDSv1 service is configured as "optional" rather than "required". When IMDSv1 is set to "optional", the EC2 instance allows access to the metadata service without requiring more secure token-based authentication, creating a security risk.

  • The query filters specifically for the EC2 RunInstances event

Query Explanation

This query detects when IAM access keys are created and then quickly deleted within a short time frame. This pattern is often associated with attackers who create temporary access keys to perform malicious actions, then delete them to cover their tracks.

Detection Logic

This KQL query identifies suspicious access key lifecycle patterns through several techniques:

  1. Time-Based Analysis:
    • Specifically looks for access keys created and deleted within a 10-minute window
    • Focuses on recent activity (within the last hour) to enable rapid response
    • Calculates the precise duration between creation and deletion events
  2. Source Filtering:
    • Leverages a custom function (AWS_filterCloudTrailCIDR) to identify operations from untrusted networks
    • This helps reduce false positives from expected administrative activities
    • Provides important context about the origin of suspicious key lifecycle events
  3. Anomaly Detection:
    • Identifies users with multiple access keys, which is against security best practices
    • Flags secondary keys that are created in addition to a user's primary key
    • Uses eye-catching visual indicators (🚨) to highlight suspicious keys
  4. Context Enrichment:
    • Adds descriptive information about both creation and deletion events
    • Includes details about the user agent to identify the tool or method used
    • Provides the AWS region and account context for each operation

Security Significance

  • Temporary access key creation and deletion is a common technique used by attackers to avoid detection
  • The short lifespan (under 10 minutes) indicates potential malicious intent rather than legitimate use
  • Activities from untrusted networks increase the likelihood of compromise
  • Secondary access keys may indicate an attacker creating credentials while maintaining existing ones
  • The user agent information may reveal unusual tools or automated processes
  • Rapid creation and deletion from the same source IP suggests deliberate activity rather than accident

Response Actions

If suspicious access key creation/deletion patterns are detected:

  1. Immediately investigate all AWS activities performed with the suspicious access key during its short lifespan
  2. Check CloudTrail logs for other suspicious actions from the same source IP address
  3. Examine the IAM user account for signs of compromise or unauthorized access
  4. Consider temporarily restricting the user's privileges until the investigation is complete
  5. If compromise is confirmed, rotate all credentials for the affected user
  6. Review IAM policies to implement stricter controls on access key creation
  7. Consider implementing automated monitoring for access key creation events
